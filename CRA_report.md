# 欧盟《网络弹性法案》（CRA）核心要点与实施指导

> **法规编号**：Regulation (EU) 2024/2847  
> **全称**：欧洲议会与欧盟理事会关于数字元素产品横向网络安全要求的法规  
> **通过日期**：2024年10月23日  
> **生效日期**：2024年12月10日  
> **全面适用日期**：2027年12月11日  

---

## 一、法案概述

《网络弹性法案》（Cyber Resilience Act，CRA）是欧盟针对**所有在欧盟市场销售的含数字元素产品**制定的横向网络安全监管法规，填补了欧盟此前对软硬件产品网络安全无统一强制要求的立法空白。

### 三大核心目标
1. 确保产品在**全生命周期**（设计→开发→生产→支持期）满足基本网络安全要求
2. 增强用户对产品网络安全属性的知情权，促进安全采购决策
3. 提升欧盟整体数字生态的网络安全韧性

---

## 二、适用范围

### 2.1 受约束的产品
**所有在欧盟市场投放的"含数字元素产品"**，包括：
- 硬件产品（路由器、摄像头、智能家居设备等）
- 软件产品（应用程序、操作系统、固件等）
- 含远程数据处理组件的产品（云端后台为产品必要组成时）

> **关键认定**：软件首次提供下载即视为"投放市场"，每次下载均视为独立副本上市。

### 2.2 豁免/特殊规则产品
| 类别 | 处理方式 |
|------|----------|
| 医疗设备、体外诊断器械 | 已被 MDR/IVDR 覆盖，豁免CRA |
| 汽车及零部件（型式认证） | 豁免CRA，遵从汽车法规 |
| 航空、航运、铁路 | 豁免CRA，遵从行业专项法规 |
| 纯开源软件（无商业化）| 不属于"市场投放"，不承担完整义务 |
| 开源软件管理者（Steward）| 承担轻量级义务（安全政策+配合监管）|

### 2.3 开源软件特别规则
- **纯开源、不收费、不商业化** → 不属于CRA完整监管范围
- **引入开源组件的商业产品制造商** → 须对开源组件进行尽职调查
- **开源软件管理者（Steward）新概念** → 需制定安全政策、配合市场监管机构

```
2024年12月10日  ←── 法案正式生效
       ↓

       ↓

```

## 四、产品分类与合规评估路径

CRA的监管对象可概括为三层：**普通产品**、**重要产品**、**关键产品**。其中：
- **附件III**列的是**重要产品**，并进一步分为 **Class I** 和 **Class II**；
- **附件IV**列的是**关键产品**。

这里按前文提到的产品直接列示，并用一句话说明对应合规路径：

| 产品类别 | 前文列举的典型产品 | 简要合规路径 |
|----------|--------------------|--------------|
| 普通产品 | 一般消费电子、通用软件应用 | **Module A（内部控制）**，制造商自评，无需公告机构 |
| 重要产品 Class I | 身份管理与特权访问管理、独立/嵌入式浏览器、密码管理器、杀毒/反恶意软件、VPN产品、网络管理系统、SIEM系统、启动管理器、PKI与数字证书软件、物理/虚拟网络接口、操作系统、路由器/调制解调器/交换机、含安全功能的芯片、智能家居虚拟助手、智能家居安全设备、联网玩具、健康监测可穿戴设备 | **完整采用协调标准/共同规范/适用认证方案时可走 Module A；否则走 Module B+C 或 Module H，需要公告机构** |
| 重要产品 Class II | 虚拟化管理程序与容器运行时、防火墙、IDS/IPS系统、防篡改微处理器、防篡改微控制器 | **走 Module B+C 或 Module H，需要公告机构** |
| 关键产品 | 带安全盒的硬件设备、智能电表网关及其他高级安全用途设备、智能卡或类似设备 | **走 Module B+C 或 Module H，需要公告机构** |

### 4.1 普通产品（默认类别）
- 代表产品：一般消费电子、通用软件应用
- 简要路径：**Module A（内部控制）**，制造商可自评，无需公告机构

### 4.2 重要产品（附件III）

#### Class I
前文列举的产品包括：
- 身份管理与特权访问管理
- 独立/嵌入式浏览器
- 密码管理器
- 杀毒/反恶意软件
- VPN产品
- 网络管理系统
- SIEM系统
- 启动管理器（Boot managers）
- PKI与数字证书软件
- 物理/虚拟网络接口
- 操作系统
- 路由器/调制解调器/交换机
- 含安全功能的芯片
- 智能家居虚拟助手
- 智能家居安全设备
- 联网玩具（含社交/位置跟踪）
- 健康监测可穿戴设备

简要路径：**完整采用协调标准/共同规范/适用认证方案时，可走 Module A；否则走 Module B+C 或 Module H，需要公告机构。**

#### Class II
前文列举的产品包括：
- 虚拟化管理程序与容器运行时
- 防火墙、IDS/IPS系统
- 防篡改微处理器
- 防篡改微控制器

简要路径：**走 Module B+C 或 Module H，需要公告机构。**

### 4.3 关键产品（附件IV）
前文列举的产品包括：
- 带安全盒的硬件设备
- 智能电表网关及其他高级安全用途设备
- 智能卡或类似设备

简要路径：**走 Module B+C 或 Module H，需要公告机构。**

## 五、附件I：基本网络安全要求（核心合规标准）

### 5.1 第一部分：产品属性要求（13项）

| # | 要求 | 关键说明 |
|---|------|----------|
| 1 | **无已知漏洞** | 上市时不得存在已知可被利用的漏洞 |
| 2 | **默认安全配置** | 以安全默认状态上市，提供重置功能 |
| 3 | **安全更新机制** | 支持自动更新（可退出），安全及时地推送补丁 |
| 4 | **防未授权访问** | 通过认证/IAM防止未授权访问，记录访问尝试 |
| 5 | **数据保密性** | 使用加密等先进技术保护存储/传输/处理中的数据 |
| 6 | **数据完整性** | 防止数据/命令/程序被篡改，报告损坏情况 |
| 7 | **数据最小化** | 仅处理产品目的所必要的数据 |
| 8 | **功能可用性** | 核心功能在故障后仍可用，具备抗DDoS能力 |
| 9 | **最小化负面影响** | 减少对其他设备/网络的负面影响 |
| 10 | **限制攻击面** | 设计上限制外部接口等攻击向量 |
| 11 | **减轻事件影响** | 采用漏洞利用缓解技术降低攻击影响 |
| 12 | **安全日志记录** | 记录安全相关活动，提供用户退出机制 |
| 13 | **安全数据清除** | 提供安全永久删除数据的功能 |

### 5.2 第二部分：漏洞处理要求（8项）

| # | 要求 | 关键说明 |
|---|------|----------|
| 1 | **识别与记录漏洞** | 制作 **SBOM（软件物料清单）**，至少包含顶层依赖项 |
| 2 | **及时修复漏洞** | 按风险优先级处理，安全更新应与功能更新分离 |
| 3 | **定期安全测试** | 对产品进行定期安全测试与审查 |
| 4 | **披露漏洞信息** | 修复后公开发布漏洞信息（描述/影响/严重性/修复指南） |
| 5 | **协调漏洞披露政策** | 建立并执行 CVD（协调漏洞披露）政策 |
| 6 | **建立漏洞报告渠道** | 提供安全漏洞报告联系方式 |
| 7 | **安全分发更新** | 建立安全机制分发补丁，安全更新应支持自动推送 |
| 8 | **免费及时更新** | 安全更新需免费提供（与商业用户另有约定除外） |

## 六、报告义务

### 6.1 三阶段报告时限

```
发现漏洞被主动利用 / 严重安全事件
         ↓
    ≤ 24小时
  早期预警（Early Warning）
  通知ENISA统一报告平台
  注明是否怀疑存在恶意行为
         ↓
    ≤ 72小时
  通知报告（Notification）
  初步评估（影响/严重性）
  如已知，提供利用指标IoC
         ↓
    ≤ 14天
  最终报告（Final Report）
  漏洞详情 + 修复方案 + 全面评估
  （复杂情况可延至1个月）
```

### 6.2 报告渠道
- 统一通过 **ENISA（欧盟网络安全局）** 建立的集中报告平台提交
- 报告将由ENISA转发至相关成员国CSIRT和主管机构

## 七、支持期（Security Support Period）

### 核心规则
- **最低5年**，但需根据产品**预期使用寿命**合理确定（生命周期短于5年的产品，支持期≥实际使用寿命）
- 须在产品上市时**明确标注支持期结束日期**（消费者知情权）
- 支持期内必须持续提供安全更新

### 重大修改的判定（影响支持期重算）
需重走完整合规流程的"重大修改"：
- ✅ 改变产品预期用途
- ✅ 引入新威胁向量或攻击场景
- ✅ 显著改变风险概率或影响
- ❌ 安全修复更新**通常不属于重大修改**
- ❌ 不影响安全合规性的功能迭代通常不属于重大修改

## 八、经济运营者义务划分

| 主体 | 核心义务 |
|------|----------|
| **制造商** | 风险评估 → 满足附件I要求 → 技术文档 → CE标志 → EU符合性声明 → 漏洞管理 → 报告义务 |
| **进口商** | 验证产品符合CRA → 检查CE标志和文档 → 发现问题采取纠正措施 |
| **分销商** | 验证CE标志和说明书 → 发现问题时通知制造商/进口商或主管机构 |
| **开源Steward** | 制定安全政策 → 配合市场监管 → 无强制技术要求 |

---

## 九、CE标志与合规声明

产品满足所有要求后，须：
1. 出具 **EU符合性声明（DoC）** — 按附件V/VI模板
2. 在产品上标贴 **CE标志**
3. 准备并保存 **技术文档（附件VII）**，包括：
   - 产品描述与预期用途
   - 网络安全风险评估
   - 符合的标准和规范清单
   - 符合性评估报告
   - SBOM（软件物料清单）

## 十、处罚机制

| 违规类型 | 最高罚款 |
|----------|----------|
| 违反基本网络安全要求（附件I）| **1500万欧元** 或全球年营业额的 **2.5%**（取较高者）|
| 违反其他义务（报告、文档等）| **1000万欧元** 或全球年营业额的 **2%**（取较高者）|
| 向市场监管机构提供虚假信息 | **500万欧元** 或全球年营业额的 **1%**（取较高者）|

> 💡 微型和小型企业（按EU定义）在某些报告义务上享有豁免。

## 十一、实施路线图（企业合规行动建议）

### 阶段一：立即行动（2024年12月 — 2025年底）
- [ ] **产品摸底**：盘点所有在欧盟销售或计划销售的含数字元素产品
- [ ] **分类判定**：对照附件III、附件IV判断产品类别（普通 / 重要〔Class I、Class II〕/ 关键）
- [ ] **GAP分析**：对照附件I的21项要求进行差距分析
- [ ] **成立合规小组**：指定产品安全负责人，建立跨部门合规机制

### 阶段二：关键准备（2026年）
- [ ] **建立SBOM能力**：为所有产品建立软件物料清单（至少顶层依赖项）
- [ ] **建立漏洞管理流程**：建立CVD政策、漏洞报告渠道
- [ ] **对接ENISA报告平台**：在2026年9月11日前完成对接测试
- [ ] **准备技术文档**：按附件VII要求准备完整技术文档
- [ ] **重要/关键产品**：Class I先判断能否以协调标准/共同规范覆盖后走自评；Class II与关键产品尽早启动公告机构评估（Module B+C或Module H）

### 阶段三：全面达标（2027年12月11日前）
- [ ] **完成所有产品的符合性评估**
- [ ] **贴附CE标志，签署EU符合性声明**
- [ ] **明确标注所有产品的支持期结束日期**
- [ ] **建立持续漏洞监测与响应机制**
- [ ] **供应链管理**：对第三方组件和开源依赖完成尽职调查

## 十二、与其他法规的衔接

| 法规 | 交叉处理原则 |
|------|--------------|
| **NIS2指令** | CRA关注产品安全，NIS2关注组织运营安全，互补不重叠 |
| **AI法案（EU 2024/1689）** | 高风险AI系统若属于CRA重要/关键产品，须同时满足两套要求 |
| **医疗器械法规（MDR/IVDR）** | MDR/IVDR覆盖的医疗设备豁免CRA |
| **汽车型式认证法规** | 汽车专用组件豁免CRA；通用渠道销售须适用CRA |
| **无线电设备指令（RED）** | 部分产品重叠，CRA将优先适用于RED的网络安全条款 |
| **通用数据保护条例（GDPR）** | 数据最小化等要求与GDPR协同，不重复处罚 |
